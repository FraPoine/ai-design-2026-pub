Course Project

Build and evaluate a large-scale AI system (Weeks 6-12)

Overview

The course project is the capstone experience of this course. You'll apply everything you've learned to design, implement, and rigorously evaluate an AI-powered system. Projects are individual for both tracks.

Project Options (Track A)

For Track A, choose one of the four project directions described in the Project Descriptions document. Each project defines a problem space, baseline capabilities, and evaluation challenges — you shape the rest.

1. Peer Review System

Multi-agent review pipeline providing structured, multi-perspective feedback on written work. Reviewer diversity, inter-rater agreement, iterative improvement.

2. Social Network for Agents (and Humans)

Social platform where AI agents replicate human personas and interact. Persona consistency, mode collapse detection, emergent social dynamics.

3. Conversational Clustering

AI agent that iteratively clusters data through dialogue with a human. Contradictory feedback, soft assignments, evaluation without ground truth.

Project Requirements

System Requirements

  • Functional AI-powered application with clear use case
  • Integration of agents, tools, and/or memory systems
  • Appropriate error handling and logging
  • Basic observability (metrics, logs)
  • Safety considerations and guardrails where appropriate

Testing Requirements

AI projects have two kinds of tests. You need both.

  • Deterministic tests — for the parts that have right answers: data ingestion, parsing, tool execution, state management, API integration. These are standard unit/integration tests. If your system extracts facts from documents, test the extractor. If it calls tools, test that the tools do what they claim.
  • Eval suite — for LLM outputs that don't have a single right answer. These run your system on your golden dataset and score the results (via LLM judge, rubric, or human labels). They won't pass/fail cleanly — they produce scores and distributions. That's fine. They should still be automated and runnable.

Concretely:

  • Use pytest. All tests runnable with a single command: uv run pytest
  • Set up GitHub Actions to run your deterministic tests on every push. A broken push should be visible immediately.
  • The eval suite can be a separate command (e.g., uv run pytest tests/eval/ --slow) since it costs money and takes time — but it must be scripted, not manual.
  • Write tests as you build features, not after. If you add a tool, add a test for it in the same commit.

Evaluation Requirements

  • Golden Dataset: Curated test set with ground truth labels
  • Metrics: Well-justified selection of evaluation metrics
  • Experiments: At least one ablation study or comparative experiment
  • Error Analysis: Systematic categorization of failure modes
  • Uncertainty: Confidence intervals or other uncertainty measures

Weekly Progress

I expect visible progress every week, and I will check. Your GitHub repository is not just a final deliverable — it is a record of how you work.

  • Commit and push work at least weekly from Week 6 onward. I review commit history.
  • Weeks with no activity will be noticed and will count against you.
  • If you're stuck, commit what you have — a failed experiment, a write-up of what you tried, updated evaluation notes. Silence is worse than a dead end.
  • This is not about volume. Three thoughtful commits that show you wrestled with a design decision beat fifty auto-generated files.
  • Use GitHub Issues to track your work. Break your project into tasks, create issues, reference them in commits (fixes #3), close them as you go. Aim for ~3–5 active issues per week. An issue stuck "in progress" for three weeks with no linked commits tells a story too.

A project that materializes fully formed in the last week tells me nothing about how you think, debug, or respond to problems — and it's usually worse than one built iteratively.

Deliverables

Both tracks: Exam

The exam for both tracks is a presentation followed by a discussion. You present your system, design choices, evaluation, and results, then answer questions.

Both tracks: Final deliverables

  • Presentation + discussion — system demo, design choices and alternatives, evaluation results, limitations
  • Report — problem statement, architecture, experiments, results, error analysis, lessons learned
  • Code repository — documented source, README with setup instructions, evaluation scripts, golden dataset

Track A: In-class milestones

  • Project spec v1 — problem statement, architecture sketch, evaluation plan
  • Review #1 — refined scope, detailed architecture, golden dataset status
  • Working sessions — implementation, evaluation runs, design clinics
  • Mid-project presentation (10–12 min internal demo)
  • Results review and presentation rehearsal
  • Final presentation + discussion (graded)

Track B: Independent milestones

  • Scope proposal — submit problem statement and high-level plan for instructor approval
  • Independent work
  • Presentation + detailed discussion (graded)

Grading Rubric (0–30)

Both tracks use the same rubric. Each dimension is scored 0–10.

Dimension (0–10 each) What we look for
Design & Prototypes Clear goals, meaningful design alternatives, working prototype(s) that reflect the design choices
Experiments & Analysis Experiments comparing variants, appropriate metrics, honest analysis of results and failures, reflection on limitations
Communication Well-structured presentation and report, clear explanation of choices and lessons learned, good Q&A
